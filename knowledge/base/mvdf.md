# MVDF — Marketing Value Delivery Framework（v0.1）

## 定義

**MVDF** は、マーケティング活動を、属人的な施策集ではなく、**顧客価値の創造・伝達・提供・交換を統合的にマネジメントするためのフレームワーク**である。PMBOK（Project Management Body of Knowledge）のマーケティング版として着想したが、最終的に「Value Delivery（価値提供）」を中心軸に据えた独自フレームワークとして定義する。

土台となる定義（AMA, 2017）:

> マーケティングとは、顧客・クライアント・パートナー・社会にとって価値ある提供物を創造し、伝達し、届け、交換するための活動・制度・プロセスである。

MVDF はこの定義の「**創造 → 伝達 → 提供 → 交換**」を、原則・領域・ライフサイクル・プロセス・成果物の 5 階層で標準化する。広告・販促だけでなく、**価値創造から市場実装、測定、学習までの全工程**を扱う。

中心コンセプト:

> マーケティングとは、市場と顧客を理解し、価値提案を設計し、適切な接点を通じて需要と関係性を創出し、事業成果と社会的信頼を継続的に高める**価値提供（Value Delivery）のマネジメント活動**である。

## SURF との関係

RMCStack の中核は **SURF（Syncing → Understanding → Releasing → Fitting）** というマーケティング OS である。MVDF と SURF は競合せず、**役割が違う**。

| | MVDF | SURF |
|---|------|------|
| 性格 | フレームワーク（What） | オペレーティング・サイクル（How） |
| 問い | 何を管理し、何を成果物として残すか | どう情報を共有・理解・解放・適合させるか |
| 単位 | 領域（Performance Domains）／成果物 | 段階（Stages）／還流ループ |
| 起点 | マーケティングの定義と原則 | チーム内の前提共有 |
| 終端 | 8 領域すべての継続的成熟 | 次サイクルの Syncing 起点 |
| 主体 | 組織（経営〜現場） | 組織 ↔ AI |

**使い分け**:
- **MVDF は地図**: 「自分たちのマーケティング活動は、どの領域をカバーできていて、どこが空白か」を点検する道具。
- **SURF は乗り物**: 「いま地図のどこにいて、次にどこへ動くか、結果をどう還流するか」を駆動する道具。

MVDF の 8 領域は、SURF サイクルの **Syncing** で点検対象として、**Understanding** でレンズとして、**Fitting** で成果物の置き場として機能する。詳細は本文末「RMCStack 上での使い方」。

## 8 つの原則

PMBOK が冒頭に「原則」を置く流儀に倣う。マーケティング判断の基本思想。

### 原則 1：顧客価値を起点にする
マーケティングは「売りたいものを売る活動」ではなく、**顧客が価値を感じ、選び、使い続ける理由を設計する活動**である。

### 原則 2：市場を仮説ではなく証拠で理解する
顧客・競合・カテゴリ・代替手段・購買行動を、調査・観察・データ・実験で**継続的に**理解する。一度作ったペルソナを更新しないのは原則違反。

### 原則 3：戦略と実行を分断しない
STP・ブランド・価格・チャネル・キャンペーン・営業連携・CX を**一貫したシステム**として扱う。戦略チームと戦術チームが分かれて噛み合わないのはここの違反。

### 原則 4：ブランドは約束であり、体験で検証される
ブランドはロゴやトーンではなく、**顧客が接触するすべての体験を通じて形成される信頼資産**。広告のキレイさより接点全体の整合性。

### 原則 5：成果は売上だけでなく、将来価値で見る
短期のリード数や CPA だけでなく、LTV・継続率・ブランド想起・顧客満足・紹介・カテゴリ内地位も見る。

### 原則 6：学習速度を競争優位にする
完璧な計画より、**仮説 → 実験 → 測定 → 学習 → 改善**のサイクルを速く回す。SURF の Fitting → 次 Syncing がここに対応。

### 原則 7：倫理・プライバシー・社会的信頼を守る
誤認・過剰訴求・ダークパターン・データの不適切利用を避け、**長期的な信頼を優先する**。

### 原則 8：マーケティングは部門ではなく組織能力である
マーケ部だけで完結せず、**経営・商品・営業・CS・広報・データ・法務**と連携して成果を出す。

## 8 つのパフォーマンス領域

PMBOK の Performance Domains に相当する、**継続的にマネジメントすべき領域**。AMA 定義の「創造・伝達・提供・交換」に紐付けて整理する。

| # | 領域 | 価値提供フェーズ | 主な問い | 主な成果物 |
|---|------|----------------|---------|-----------|
| 1 | **市場・顧客理解** | （前提） | 誰のどんな問題を扱うのか | 市場分析、顧客インサイト、ペルソナ、JTBD |
| 2 | **戦略・ポジショニング** | 創造 | どこで勝つのか | STP、ポジショニング、価値提案 |
| 3 | **ブランド・コミュニケーション** | 伝達 | 何者として記憶されるか | ブランド戦略、メッセージ、トーン&マナー |
| 4 | **提供価値・商品連携** | 創造 | 何をどう価値化するか | 価値仮説、価格方針、パッケージ設計 |
| 5 | **チャネル・顧客接点** | 提供 | どこで出会い、どう体験させるか | チャネル戦略、カスタマージャーニー |
| 6 | **需要創出・キャンペーン** | 伝達 + 提供 | どう認知・興味・検討・購買を生むか | キャンペーン計画、コンテンツ設計、メディア計画 |
| 7 | **顧客関係・成長** | 交換 | どう継続・拡張・紹介につなげるか | CRM、LTV 施策、リテンション計画 |
| 8 | **測定・ガバナンス** | （横断） | 成果をどう判断し、改善するか | KPI ツリー、ダッシュボード、予算管理、リスク管理 |

8 領域はすべて**並行して動く**（プロジェクトのフェーズではない）。あるキャンペーンが進行中でも、市場理解は更新され続け、ブランドは検証され続ける。

## 6 段階ライフサイクル

PMBOK のプロジェクト・ライフサイクルに相当。マーケティング・イニシアチブ単位の流れ。

```
Diagnose → Define → Design → Deliver → Measure → Adapt
診断   →  定義  →  設計  →  実行  →  測定  →  適応
```

| 段階 | やること |
|------|---------|
| **Diagnose（診断）** | 市場・顧客・事業課題を診断する |
| **Define（定義）** | ターゲット・価値提案・目標を定義する |
| **Design（設計）** | 戦略・ブランド・チャネル・施策を設計する |
| **Deliver（実行）** | キャンペーン・コンテンツ・営業連携を実行する |
| **Measure（測定）** | 成果・学習・リスクを測定する |
| **Adapt（適応）** | 改善・再配分・スケール・撤退を判断する |

BtoB / BtoC / SaaS / D2C / 店舗 / 採用 / 自治体広報まで転用可能。

### SURF との重なり

| MVDF ライフサイクル | SURF 段階 |
|---------------------|-----------|
| Diagnose, Define | Syncing（前提共有） + Understanding（診断レビュー） |
| Design | Understanding + Releasing（既成枠の手放し） |
| Deliver | Fitting（shipping） |
| Measure, Adapt | Fitting（還流） → 次サイクルの Syncing |

MVDF は**個別イニシアチブの直線的な流れ**を扱い、SURF は**組織学習の循環**を扱う。

## プロセス体系（入力・技法・出力）

PMBOK の ITTO（Inputs / Tools & Techniques / Outputs）に倣う。

| プロセス | 入力 | ツールと技法 | 出力 |
|---------|------|-------------|------|
| 市場機会を評価する | 事業目標、売上データ、競合情報 | 3C、PEST、カテゴリ分析 | 市場機会仮説 |
| 顧客セグメントを定義する | 顧客データ、調査結果 | STP、クラスタ分析、JTBD | ターゲットセグメント |
| 価値提案を設計する | 顧客課題、商品情報 | バリュープロポジションキャンバス | 価値提案文 |
| ポジショニングを定義する | 競合比較、顧客認知 | ポジショニングマップ | ポジショニングステートメント |
| KPI を設計する | 事業目標、ファネルデータ | KPI ツリー、North Star Metric | KPI 体系 |
| キャンペーンを計画する | 戦略、予算、顧客接点 | メディア計画、コンテンツマップ | キャンペーン計画書 |
| コンテンツを設計する | 顧客インサイト、検索意図 | メッセージ設計、編集カレンダー | コンテンツ計画 |
| リード・商談を管理する | 問い合わせ、MA/CRM データ | スコアリング、ナーチャリング | MQL/SQL 定義 |
| 顧客維持を設計する | 利用データ、解約理由 | コホート分析、CRM 施策 | リテンション計画 |
| 成果をレビューする | KPI 実績、費用、学習ログ | 差分分析、A/B テスト、MMM | 改善提案 |

## 4 象限ビュー（実務サマリ）

8 領域を実務で動かすときの最小単位。

| 象限 | 目的 | 含める内容 |
|------|------|-----------|
| **Insight** | 市場と顧客を理解する | 調査、データ分析、インサイト、競合分析 |
| **Strategy** | 勝ち筋を決める | STP、価値提案、ブランド、価格、KPI |
| **Execution** | 市場に届ける | チャネル、広告、コンテンツ、営業連携、CRM |
| **Learning** | 成果から学ぶ | 測定、改善、実験、予算配分、ガバナンス |

## 適用例（領域別の代表的な重み）

| 業態 | 重みが大きい領域 | 補足 |
|------|----------------|------|
| **BtoB** | 1（顧客理解 / JTBD）、2（戦略）、6（需要創出 / ABM）、7（顧客関係） | 営業連携が成否を分ける。MQL → SQL 定義の整合性は領域 8 の責務 |
| **BtoC** | 3（ブランド）、5（チャネル）、6（キャンペーン）、8（測定） | ブランド資産の継続投資と短期需要創出の両立 |

| **新規事業** | 1、2、4 | まず Diagnose → Define を厚く。需要創出に走る前に Problem-Solution Fit |

## RMCStack 上での使い方

MVDF は RMCStack のスキル群と次のように対応する。**MVDF の 8 領域 × SURF 段階 × RMCStack スキル**のクロス。

### MVDF 領域 × RMCStack スキル

| MVDF 領域 | 主に使う RMCStack の道具 |
|-----------|----------------|
| 1. 市場・顧客理解 | `customer-research-jtbd.md`、`/set-workspace`（ICP 入力） |
| 2. 戦略・ポジショニング | `/cmo-review`、`/release-assumptions`、`/consultant-review` |
| 3. ブランド・コミュニケーション | `/set-brand`、`/creative-direction` |
| 4. 提供価値・商品連携 | `pricing-strategy.md`、`/cmo-review` |
| 5. チャネル・顧客接点 | `cvr-optimization-playbook.md`、`/seo` |
| 6. 需要創出・キャンペーン | `digital-advertising.md`、`content-marketing.md`、`/seo` |
| 7. 顧客関係・成長 | `retention-lifecycle.md`、`cvr-optimization-playbook.md` (signup-flow / onboarding) |
| 8. 測定・ガバナンス | `web-analytics-practice.md`、`/feedback`、`/ceo-review` |

### MVDF 領域 × SURF 段階での使い方

- **Syncing で**: 8 領域を**充足チェックリスト**として使う。「うちは領域 7（顧客関係）が空白」「領域 8（測定）の KPI ツリーがない」を可視化する。
- **Understanding で**: 領域ごとのレンズで `/cmo-review` `/ceo-review` `/seo` 等を回す。
- **Releasing で**: 領域横断で**聖域 KPI**（典型: 領域 8 の MQL 数）を `/release-assumptions` で疑う。
- **Fitting で**: 成果物の置き場として領域別ディレクトリを意識する（`private/memory/workspaces/<slug>/profile/` と `results/` の中で領域別に整理）。

### MVDF ライフサイクル × RMCStack ワークフロー

| MVDF ライフサイクル | 対応する RMCStack の動き |
|---------------------|--------------------------|
| Diagnose | `/next` → `/surf-check` → `customer-research-jtbd.md` 参照のインライン調査 |
| Define | `/set-workspace` + `/cmo-review`（戦略レビュー） |
| Design | `/creative-direction` + `/release-assumptions` |
| Deliver | `content-marketing.md` / `digital-advertising.md` / `cvr-optimization-playbook.md` 参照のインライン実行 |
| Measure | `web-analytics-practice.md` 参照のインライン分析 |
| Adapt | `/feedback` → 次サイクルの `/set-update` |

## 命名の経緯

検討した候補:

| 候補 | 評価 |
|------|------|
| MMBOK: Marketing Management Body of Knowledge | PMBOK 風で本格的だが、固有性が弱く検索でも埋もれる。「知識の集積」という静的な印象 |
| MBOK: Marketing Body of Knowledge | シンプル。商標・固有性の懸念 |
| **MVDF: Marketing Value Delivery Framework** ✅ | **採択**。AMA 定義の「Value Delivery」を中心に据えられる。PMBOK の権威性に依存せず独自性が高い |
| Marketing Operating System | SURF と意味が重なるため不採用 |
| Marketing Excellence Framework | 認定制度寄りでマーケティング実務体系として広すぎる |

**RMCStack 内部呼称**: **MVDF**。日本語は「マーケティング価値提供フレームワーク」または「価値提供標準」。

「Body of Knowledge（知識体系）」ではなく「Framework」を選んだ理由:
- 知識体系は**蓄積物**として静的に響くが、MVDF は SURF と組み合わせて**動かす**ことを前提とする
- 「Value Delivery」を冠することで、AMA 定義との整合と、施策集ではなく価値提供のマネジメントであるという主張が同時に立つ
